How The Run Index Was Built

Summary of the process used to create the dated run index, pair related pages, extract the best variants, render preview images, and publish the final summary page.

The final deliverable was a grouped index page for the HTML runs in C:\Users\toont\Documents\GitHub\misc, covering files from 03/04/2026 onward. Each entry links back to the original run pages and shows a rendered image of the highest-scoring turtle program.

1. Finding The Source Files

The first step was to scan the misc folder for HTML files created on or after 03 April 2026. The naming was not fully consistent, so the selection could not rely on one rigid pattern.

  • Files were filtered by date and by timestamp-like filename fragments such as 2026-04-05T16-35-49.
  • Both newer history pages and matching dissolve pages were included.
  • Older logo-evolution pages were treated as history pages when they used the earlier format.

2. Reading The Embedded Data

The run pages were not all simple static documents. Most of the useful information was embedded inside JavaScript.

  • History pages usually contained a base64-encoded RAW payload with iteration history, variant scores, critic comments, and turtle code.
  • Dissolve pages usually contained frame arrays instead of the full RAW object.
  • A Python build script was written to decode both formats and normalize them into one shared data structure.

The script that performs this normalization is [build_index.py](C:/Users/toont/Documents/New%20project/output/misc-run-index/build_index.py).

3. Pairing Related Pages

Runs often had both a history page and a corresponding animation page. Those needed to appear as one entry.

  • Filenames were reduced to a normalized stem, removing suffixes such as history, logo-evolution, and dissolve.
  • When more than one candidate existed, the nearest timestamp was used to pair a history page with its matching animation page.
  • Unpaired outliers were still preserved as standalone entries.

4. Choosing The Best Variant

Each history page could contain many iterations and variants, so the page needed one representative image per entry.

  • All scored variants in the embedded run history were collected.
  • The highest total score was used as the primary ranking.
  • Winner flags, aesthetic score, novelty score, and later-stage preference were used as tie-breakers.
  • The selected variant's turtle program became the image source for the entry.

5. Rendering The Images

The request was to use the turtle graphics code from the most recent history page. Instead of rewriting the interpreter, the existing one was reused directly.

  • The most recent history page's browser-side turtle renderer was loaded in Playwright.
  • Each chosen turtle program was injected into that page's existing compileLogo, autoFitQueue, and renderQueue pipeline.
  • A deterministic random seed was used during export so stochastic programs produced stable images.
  • Each entry was exported as a moderate-size PNG, and clicking the thumbnail opens the full-size image.

6. Building The Final Page

After the data and images were ready, the summary page itself was generated as HTML.

  • Entries were grouped by taste.
  • Each card included a cleaned-up title, date, concise run metadata, score details, critic comment, and links to the source pages.
  • Redundant phrasing was removed in a later refinement pass.
  • The label dissolve was renamed to animation of winners.

The published index is [2026-04-03-run-index.html](C:/Users/toont/Documents/GitHub/misc/2026-04-03-run-index.html).

7. Today's Additions

The index was updated again on 08 April 2026 with seven new entries and a few extra presentation rules to make the cards more informative.

  • The five prompt-experiment runs now carry short explanatory notes, with the quoted prompt changes preserved but reworded for clarity.
  • Runs that changed mutation magnitude during the history now show a concise timeline of the shift inside the entry.
  • Two flower runs begin from custom seed programs rather than FD 100, so those entries now show both the starting image and the winning image.
  • The page now contains 38 grouped entries drawn from 70 dated source files.

8. Verifying The Result

Before finishing, the output was checked in two ways.

  • The generated JSON and image folder were checked to make sure every entry had a matching PNG, plus two extra starting images for the custom-seed flower runs.
  • The final deployed HTML in misc was loaded in the browser to confirm card count, image references, link labels, and the additional notes.
The final build now produces 38 grouped entries from 70 dated source files, with 40 rendered images overall, including the two custom starting images.
